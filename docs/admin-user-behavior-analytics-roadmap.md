## 功能定位

`用户行为分析` 面向管理员，用于从全局与单个用户两个层面观察学习记录、行为事件、学习会话、活跃变化、表现波动和异常风险。现有 `统计分析` 继续服务当前登录用户，新页面不复制个人统计，而是提供全体用户聚合、用户列表、用户详情、行为事件、会话分析和导出。

统一称谓：

- 管理端角色称为：管理员
- 被统计对象称为：用户
- 页面、图表、表格、导出字段、空状态和错误提示中统一使用“用户”

## 指标口径

| 指标 | 口径 |
| --- | --- |
| 管理员 | 当前登录用户存在于 `admin_users` |
| 有效记录 | `deleted_at_ms` 为空的 `study_records` |
| 活跃用户 | 统计周期内存在有效学习记录；事件/会话上线后可扩展为存在有效事件或有效会话 |
| 沉默用户 | 历史上存在记录，但统计周期内没有有效记录 |
| 最近活跃时间 | 最近一次有效记录创建或更新事件时间 |
| 平均正确率 | 练习记录中 `correctQuestions / totalQuestions` 汇总计算 |
| 连续活跃天数 | 基于有效记录的 `record_date` 连续日期计算 |
| 删除记录数 | `deleted_at_ms` 非空的记录数量 |
| 页面访问次数 | `analytics_events.event_name = 'page_view'` |
| 学习时长 | `learning_sessions.active_duration_ms` |

## 数据与权限

- `admin_users`：手动维护管理员身份，前端不提供自助成为管理员入口。
- `analytics_events`：记录页面访问、记录增删改、导入导出、同步、错误等真实行为事件。
- `learning_sessions`：记录一次连续使用过程，区分总停留时长、有效活跃时长和空闲时长。
- 管理员 RPC 必须先校验 `auth.uid()` 是否在 `admin_users` 中。
- 普通用户只能读写自己的记录、事件和会话。
- 本地模式不支持全体用户统计，只能展示当前用户样例，并明确提示统计口径限制。

## 页面结构

`/admin/analytics` 提供单页多 Tab 管理台：

- 总览：KPI、每日趋势、记录结构、风险用户
- 用户：筛选、排序、用户级统计列表
- 记录：题型、素材、总结结构
- 行为：功能使用、行为漏斗、错误事件、用户时间线
- 会话：有效时长、会话趋势、活跃时段热力图、会话列表
- 导出：当前筛选用户列表、记录明细、异常用户、行为事件、单个用户报告

`/admin/users/:userId` 复用同一页面的数据能力，聚焦单个用户概览、学习趋势、题型表现、素材与总结、行为时间线和会话记录。

## 易误判边界

- 不能用 `created_at_ms` 到 `updated_at` 推断学习时长。
- 不能把“有记录”直接等同于真实使用。
- 不能把记录数量直接等同于学习质量。
- 删除记录不一定是负向行为，可能只是纠错。
- `record_date` 可能是用户填写日期，不一定是真实发生时间。
- 本地未同步数据不能计入全体用户统计。
